How Open Algorithms can Solve Identity and Data-Sharing Problems?

What if we can have all data in one place, share data in a privacy-preserving way, or even have distributed machine learning? Thanks to Prof. Alex Pentland and Dr. Thomas Hardjono by inventing the Open-Algorithms and making it possible :) and as it’s so interesting, I thought to summarize the paper.

The value that data can provide is unbelievable and it’s proving great value to a lot of domains and will have more impact in the future of digital society but until now less than 1% of the world's data is analyzed, data sharing gives great benefits and more insights if you have the data from multiple domains but the data-sharing ecosystem faces a lot of challenges, like:

  • Sharing data outside the organization is so hard.
  • Accessing personal-data is not efficient because of law and technical infrastructure.
  • GDPR had a very big impact on global organizations that rely on the Internet for the trans-border flow of raw data.
  • Operate with a native asset, without a primary ledger.

The identity problem in reality a data-sharing problem for that reason if we want to understand the solution provided we have to a brief history and overview of identity management and federation but first I am going to go through the main problems in the identity and access management ecosystem:

  • The biggest problem is that identity services are tied to specific services.
  • Duplication is a very big problem, for ex: all services hold the same person-attributes.
  • In most cases, users don’t have an idea of what data is collected and what algorithms companies are running on it!
  • There is a big diminishing of trust in data holders.
  • Some service providers only have access to only poor quality user data.

So mainly the start of the need for identity providers is when a lot of people wanted to have internet services and the solution was inventing the Web-SSO which is built on the idea of having a trusted third-party that can be considered as an identity-provider and to have standardization the Liberty Alliance Project started which included over 150 companies and organizations that resulted:

  • SAML2.0: basically, it is an open standard that allows an identity provider to give authorization to a service provider and uses XML for communication because the main thing is separating the identity provider from the services provider and it remains the predominant Web-SSO technology used within Enterprise.
  • Shibboleth: Shibboleth is an open-source SSO which allows you to use the same identity for multiple service providers.

The rise of web apps and mobile applications needed a new system or authentication, for example, there was a big need of allowing other apps to access your calendar without giving the password and being able to disable it after a time which was the reason why the OAuth2.0 protocol was invented but the design of it lacks details for practical implementation and deployment!

One other important thing to discuss before going into OPAL is Identity Federation, you can think about it in this way, what if you want a service but you were not known to another identity provider, the solution will be that we decentralize the identity providence and having a network of them so when an IDP doesn’t have your data it can query it from another IDP in the federation and the federation includes, Identity Provider, attribute, Attribute Provider, Identity Federation, Relying Party or Service Provide and it uses SSO as a primary flow that makes a lot of limitations which are:

  • Identity services are becoming a side function for another service.
  • Lack of high-quality data.
  • Limited user-control over the data.

So now is the time for the core of the article which is Open-Algorithms and it’s the principles and concepts of it:

  • Don’t move the data, move the algorithm to the data.
  • Don’t Export the data from its repository.
  • Analyze that the algorithms are analyzed for safety and privacy-preservation.
  • Only provide aggregate answers.
  • Develop a Trust Network for Data Sharing Federation.
  • Consent for algorithm execution.
  • Having a Decentralized Data Architectures.

There are 3 steps to apply the Query-Response Model for the OPAL:

  • Select one or more vetted algorithms from an entity you want to provide data.
  • An OPAL contract that contains among others the desired Algorithm-ID is digitally signed by the Querier and sent to the target data repository.
  • The data provider checks the identity of the Querier and executes the algorithm and the result is then placed into an “OPAL contract response”.

OPAL can also help in the problems of identity federation so instead of sharing static attributes, we can start sharing vetted algorithms and what should happen is the following:

  • Algorithms instead of attributes.
  • bringing data providers from differing domains or vertical together so you can get better knowledge and new insights.
  • Here data owners could develop new business models because they can sell the executions of the algorithms, so we should have correct pricing models.
  • Data returns should be measurable for data providers.
  • Log everything on the blockchain for transparency and regulatory compliance.

The main thing now to ask is what the trust model of the OPAL Federation looks like because, for OPAL-based information sharing federation to be developed, it should use and expand the currently existing legal trust frameworks for identity systems and it has two main components the first one is the business and technical operational rules and the second are contract-based legal rules but in general it must address the following:

  • Verifying the correct matching between an identity and the set of data in a repository.
  • Providing the correct result from an OPAL-based computation to the party that requires it to complete a transaction.
  • Maintaining and protecting the private data within repositories over its lifecycle.
  • Defining the legal framework that defines the rights and responsibilities of the parties, allocates risk, and provides a basis for enforcement.

One of the most important parts in the identity management systems is the consent management and OPAL introduces a very interesting one because here it’s different, you’re asking for safe-answers not sharing the data itself and the other party has to make sure that the data stays in the same place and the OPAL systems should at least have the following:

  • Already identified algorithms.
  • Data-sets and any copy of it should be identifiable.
  • Data-providers and queriers should be identified.
  • result of the execution of the algorithm should have terms.

As the majority of apps uses OAuth2.0, it makes sense to extend it by adding User-Managed Access instead of building another one and it includes 6 main steps:

  • The Consent Management system becomes the UMA Authorization Server (AS).
  • Match the rules with the resources.
  • The querier selects an algorithm.
  • The querier must get a signed JSON data structure.
  • The Querier binds the execution-request to the consent token and transmits it to the data provider.
  • The repository returns the response to the querier.

The future directions of the OPAL project are so wide but mainly it goes into three things, Distributed Machine Learning, fairness and accountability in machine learning, and algorithms expressed as smart contracts.

Finally, three projects are using OPAL that is cool to go through:

    OPEN TrialChain (Clinical Trials):

  • TrailChain is a blockchain-based platform to validate data integrity in large, biomedical research studies that improve the visibility of data quality and analysis.

    • SMART^2 OPAL (Urgent Care):

  • SMART^2 OPAL helps decision-makers optimize the urgent care system understanding the dynamics of time between symptom to arrival to the emergency department using timestamps.

    • RoboChain (Clinical Intervention):

  • A learning framework for secure, decentralized, computationally efficient data and model sharing among multiple robot units installed at multiple sites, and it allows access to our private data to learn from these data and improve their interaction capabilities.
